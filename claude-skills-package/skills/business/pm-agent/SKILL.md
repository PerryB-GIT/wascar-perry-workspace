# PM Agent - Project & Product Management Master Toolkit

## PM Philosophy
```
Project Management: Get the right thing DONE on time and budget
Product Management: Figure out the RIGHT THING to build

Both require: Clarity, Communication, Prioritization, Execution
```

---

## Project vs Product Management

### When to Use Which
```
PROJECT MANAGEMENT (Use: /project-management-pmp)
- Defined scope, timeline, budget
- Clear deliverables
- Temporary endeavor with end date
- Success = delivered on time, budget, scope

Examples:
- Website redesign
- Office relocation
- System implementation
- Marketing campaign
- Event planning

PRODUCT MANAGEMENT (Use: /product-management)
- Ongoing product evolution
- Discovery + delivery
- No "end date"
- Success = customer value + business outcomes

Examples:
- SaaS platform
- Mobile app
- E-commerce store
- Internal tools
- API products
```

## Project Management Overview (PMP/PMBOK)

### The 5 Process Groups
```
1. INITIATING
   - Define project
   - Identify stakeholders
   - Create project charter
   - Get authorization

2. PLANNING
   - Define scope (WBS)
   - Create schedule
   - Estimate costs/budget
   - Plan quality, resources, communications
   - Identify risks

3. EXECUTING
   - Direct and manage work
   - Acquire and develop team
   - Manage communications
   - Conduct procurements

4. MONITORING & CONTROLLING
   - Track progress
   - Control scope, schedule, cost
   - Monitor risks
   - Report performance

5. CLOSING
   - Finalize deliverables
   - Obtain acceptance
   - Close contracts
   - Document lessons learned
```

## Prioritization Frameworks

### RICE Scoring
```
R - Reach: How many users affected?
I - Impact: How much will it affect them? (0.25, 0.5, 1, 2, 3)
C - Confidence: How sure are we? (0-100%)
E - Effort: Person-months to complete

SCORE = (Reach Ã— Impact Ã— Confidence) / Effort

Example:
Reach: 10,000 users
Impact: 2 (high)
Confidence: 80%
Effort: 2 person-months

Score = (10,000 Ã— 2 Ã— 0.8) / 2 = 8,000
```

### MoSCoW Method
```
M - Must Have: Critical for launch/success
S - Should Have: Important but not critical
C - Could Have: Nice to have if time permits
W - Won't Have: Out of scope (for now)

Use for: Release planning, MVP scoping
```

### Kano Model
```
MUST-BE (Basic): Expected, no delight if present, dissatisfaction if absent
- Example: Login works, pages load

PERFORMANCE (Linear): More is better, satisfaction scales with delivery
- Example: Page speed, storage space

DELIGHTERS (Excitement): Unexpected features that create joy
- Example: Surprise animations, smart suggestions

Use for: Feature classification, customer expectations
```

## Agile/Scrum Quick Reference

### Scrum Roles
```
PRODUCT OWNER:
- Owns the backlog
- Defines priorities
- Accepts/rejects work
- Voice of customer

SCRUM MASTER:
- Facilitates ceremonies
- Removes impediments
- Coaches the team
- Protects the sprint

DEVELOPMENT TEAM:
- Self-organizing
- Cross-functional
- Delivers increment
- Estimates work
```

### Scrum Ceremonies
```
SPRINT PLANNING (Start of sprint)
- What: Select backlog items for sprint
- Who: Whole team
- Duration: 2-4 hours for 2-week sprint

DAILY STANDUP (Daily)
- What: Quick sync on progress/blockers
- Who: Dev team (+ optional observers)
- Duration: 15 minutes max

SPRINT REVIEW (End of sprint)
- What: Demo completed work
- Who: Team + stakeholders
- Duration: 1-2 hours

RETROSPECTIVE (End of sprint)
- What: Reflect and improve
- Who: Scrum team
- Duration: 1-1.5 hours
```

### Sprint Metrics
```
VELOCITY: Story points completed per sprint
- Use for: Capacity planning, forecasting

BURNDOWN: Work remaining over time
- Use for: Sprint progress tracking

CYCLE TIME: Time from start to done
- Use for: Process efficiency

LEAD TIME: Time from request to delivery
- Use for: Customer responsiveness
```
